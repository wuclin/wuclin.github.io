SpringCloud

SpringCloud和SpringBoot关系

  • SpringCloud依赖于SpringBoot
  • SpringBoot主要用于快速开发单个个体微服务（可以把每一个服务打包成jar包）
  • SpringCloud主要用于全局微服务协调管理

注意使用时，版本要对应

Dubbo和SpringCloud对比

最大区别：SpringCloud抛弃了Dubbo的RPC通信，采用的是基于HTTP的REST方式

Dubbo的定位是RPC框架，SpringCloud的是微架构服务的一站式解决方案

Dubbo主要用于分布式，SpringCloud主要用于微服务了。

Nginx和Ribbon

Nginx是一种集中式的负载均衡器。简单理解就是 将所有请求都集中起来，然后再进行负载均衡。使系统达到高可用，把用户请求分配到多个服务上

而对于 Ribbon 来说它是在消费者端进行的负载均衡。

在 Nginx 中请求是先进入负载均衡器，而在 Ribbon 中是先在客户端进行负载均衡才进行请求的。

不再用端口号去访问，而是用请求会根据负载均衡算法自己选择服务

Feign负载均衡

feign是声明式的web service客户端，让微服务之间的调用变得简单，类似controller调用service。

使用Ribbon + RestTemplate时，使用RestTemplate对Http请求封装处理，形成模板化调用。

而使用Feign，只需要创建一个接口并使用注解，类似于Dao接口上的@Mapper，在为服务接口上的是@Feign注解

Hystrix （熔断和降级）

在分布式环境中，不可避免地会有许多服务依赖项中的某些失败。Hystrix是一个库，可通过添加等待时间容限和容错逻辑来帮助您控制这些分布式服务之间的交互。Hystrix通过隔离服务之间的访问点，停止服务之间的级联故障并提供后备选项来实现此目的，所有这些都可以提高系统的整体弹性。

例子：

熔断

在服务端，某个服务器超时或异常，引起熔断

整个微服务系统是这样的。服务A调用了服务B，服务B再调用了服务C，但是因为某些原因，服务C顶不住了，这个时候大量请求会在服务C阻塞。

服务C阻塞了还好，毕竟只是一个系统崩溃了。但是请注意这个时候因为服务C不能返回响应，那么服务B调用服务C的的请求就会阻塞，同理服务B阻塞了，那么服务A也会阻塞崩溃。造成服务雪崩

所谓 熔断 就是服务雪崩的一种有效解决方案。当指定时间窗内的请求失败率达到设定阈值时，系统将通过 断路器 直接将此请求链路断开。

也就是我们上面服务B调用服务C在指定时间窗内，调用的失败率到达了一定的值，那么 Hystrix 则会自动将 服务B与C 之间的请求都断了，以免导致服务雪崩现象。

其实这里所讲的 熔断 就是指的 Hystrix 中的 断路器模式 ，你可以使用简单的 @HystrixCommand 注解来标注某个方法，这样 Hystrix 就会使用 断路器 来“包装”这个方法，每当调用时间超过指定时间时(默认为1000ms)，断路器将会中断对这个方法的调用。

微服务网关–Zuul

包括请求路由和过滤两个主要功能，最终会注册进Eureka。实现外部访问统一入口

ZUUL 是从设备和 web 站点到 Netflix 流应用后端的所有请求的前门。作为边界服务应用，ZUUL 是为了实现动态路由、监视、弹性和安全性而构建的。它还具有根据情况将请求路由到多个 Amazon Auto Scaling Groups（亚马逊自动缩放组，亚马逊的一种云计算方式） 的能力

在上面我们学习了 Eureka 之后我们知道了 服务提供者 是 消费者 通过 Eureka Server 进行访问的，即 Eureka Server 是 服务提供者 的统一入口。那么整个应用中存在那么多 消费者 需要用户进行调用，这个时候用户该怎样访问这些 消费者工程 呢？当然可以像之前那样直接访问这些工程。但这种方式没有统一的消费者工程调用入口，不便于访问与管理，而 Zuul 就是这样的一个对于 消费者 的统一入口。

可以不用老是用locahost:8080,localhost:8081的形式去访问

Config

每个服务都有自己的配置文件，使用Spring Cloud Config作为一个独立的微服务应用去管理配置信息。

通过服务端可以读到目的的配置文件，可以是本地仓库也可以是远程仓库
