线程交互

Posted on 2022-06-08 In Java

同步队列和等待队列

对应线程的两个状态，阻塞和等待

阻塞：因为synchronized没获取到锁的线程就会阻塞在同步队列，同步队列中放的是尝试获取锁但失败的线程

等待：使用了wait方法，就会进入到等待队列，等待进入同步队列去竞争同步锁，进入等待队列之后，需要notify唤醒或者使用wait(long timeout)等待指定时间后唤醒

等待队列中会有多个线程wait同一个对象的情况，此时如果某一个线程调用了对象的notify方法，会任意的唤醒等待队列中的一个线程

synchronized只有一个等待队列，一个同步队列

lock锁下有多个等待队列，一个同步队列

同步队列是在同步环境下才有的概念，一个对象（即一个锁）只对应一个同步队列

由于synchronized只有一个等待队列，所以在使用notify时，只能从同步队列中随机唤醒线程

不同于synchronized，AQS的等待队列是有多个的，等待队列用来保存被阻塞的线程，在condition中，有多少个condition就对应多少个等待队列，这样就可以将获得锁的线程指定放入哪一个等待队列中，达到精准通知和等待

死锁

死锁：两个或两个以上的进程（或线程）在执行过程中，因争夺资源而造成的一种互相等待的现象

死锁产生条件：

1.互斥等待（有锁）：一个资源一次只能被一个进程使用

2.持有锁并等待锁：一个进程因请求资源而阻塞时，对已获得的资源保持不放

3.循环等待：若干进程之间形成一种头尾相接的环形等待资源关系

4.无法剥夺的等待（有些锁有超时功能）：进程获得的资源，在未完全使用完之前，不能强行剥夺

tips：解决死锁只需要破除其中一个条件即可

解决死锁的方法

  1. 破除互斥等待（一般没法破除）
  2. 破除持有并等待（一次性获取所有资源）：拿一个锁并尝试拿第二个锁，拿不到则放掉第一个锁，等一段时间重新来一遍）：程序设计的时候最好就设计好，不然以后更改很麻烦
  3. 破除循环等待（按顺序获取资源）：比如对于账户A、B，他们有ID，规定操作这两个账户的线程必须按照ID值的大小获取，这样任何线程都必须得先获得A再获得B，而不是有一个先获得A，有一个先获得B（缺点：id并不一定可以用于排序，可能得强制加上一个可以排序的规则，当以后有其他属性加入，可能引入新的id，这样又得重新为该属性分配一个id来排序）
  4. 破除无法剥夺的等待（加入超时）：不得已的方法，（缺点：用户体验不好，比如用户转账等了半天结果告诉你转账失败）

死锁例子

两个线程各自持有对方需要的资源，且同时请求对方的资源，并且都不会主动去释放资源，这样就会造成死锁

线程交互

一般线程交互有几种方式：

  1. 使用volatile变量进行控制

  2. 使用wait和notify方法

    关于wait和notify：

    首先这两个方法都要在同步块中使用，依赖的是底层的monitor，其次

    wait方法在使用的时候，对于Object.wait()来说，如果此时没有线程没有持有Object对象的锁就会报java.lang.IllegalMonitorStateException错误，同时在使用wait方法时，会立即释放Object对象的锁

    notify方法在使用的时候，用来通知那些可能等待对象的对象锁的其他线程，如果有多个线程等待，则由线程规划器随机挑选出其中一个wait状态的线程，对其发出通知notify，并使它等待获取该对象的对象锁。执行notify方法后，当前线程不会马上释放对象锁，wait状态的线程也不能马上获取对象锁，需要等执行notify的线程将程序执行完，也就是退出同步块后，当前线程才会释放锁，wait状态的线程才能获得锁。

    当第一个获得该锁的线程运行完以后，会释放掉该对象锁，此时如果没有该对象再次使用notify语句，则即使该对象已经空闲，其他wait状态的线程由于没有得到该对象的通知，还会继续阻塞在wait状态，直到这个对象发出notify或notifyAll，这里可以在使用wait(long timeout)，这样如果没有线程调用notify方法，在timeout时间之后，会自动唤醒，这里的醒过来是指线程从等待队列到了锁池队列，在获取到锁之前不会继续执行

  3. 使用join方法

    join的功能内部是使用wait方法来实现的，可以用于同步，使线程之间的并行操作变为串行操作，在A线程中调用了B线程的join方法时，表示只有当B线程执行完毕时，A线程才能继续执行。

    调用了join方法的当前线程，会进入等待队列（相当于隐式调用了wait方法），当join方法对应的线程执行完毕时，就会唤醒当前线程（当前线程离开等待队列，进入同步队列），此时要注意如果在同步队列中有多个线程在等待锁，当前线程也需要等待，同步竞争锁，也就是当前线程不一定会获得锁，得看cpu分配

    这里要注意join方法唤醒当前线程的时候

    join方法例子：

    加了b.join()，输出：
    Thread-0B

    mainA

    没有加b.join()，输出：

    mainA
    Thread-0B

    使用了join方法之后，main线程就需要在线程b运行完后在运行，这里要注意线程b要先执行start方法之后再调用join方法才有意义，如果线程没有开始，同步是没有意义的。

    关于join和wait的一个区别就是唤醒时机，后者需要在别的线程里手动唤醒（调用notify方法），前者只需要等待join执行完毕，调用这行代码的当前线程就会被唤醒（这里唤醒了不一定会获取锁）

  4. 使用await和siganl（condition条件锁）

    前面说到lock可以有多个等待队列是通过condetion实现的，调用lock.newCondition()方法创建多个condition对象，也就是一个lock可以持有多个等待队列
