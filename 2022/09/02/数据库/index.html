数据库部分

数据库三大范式

1.第一范式（1NF）：列不可再分
每一列都是不可再分的属性，确保每一列的原子性，比如地址可以拆分为省，市
2.第二范式（2NF）：属性完全依赖于主键
在第一范式的基础上，每一行数据可以被区分，通常是加多一个列，也就是主键，主要是针对于联合主键的情况，要变成只有一个主键
3.第三范式（3NF）：属性不依赖于其他非主属性，直接依赖于主键
即数据不能存在传递关系，每个属性都和主键有直接关系而不是间接关系，非主键外的所有字段必须互不依赖

索引

聚集索引，非聚集索引，联合索引，最左匹配原则，非聚集索引要回表查询，非聚集索引不回表的情况

in or关键字不走索引，like模糊匹配，前调配不走索引，后通配走索引
索引是一种数据结构，用来提高查询速度，同时会减低更新表的速度，更新表时，不仅要保存数据还要保存索引文件。
有hash索引和b+树索引，为什么没有选择hash索引？
因为hash索引不支持顺序和范围查询
B树和B+树异同：

  • B 树的所有节点既存放键(key) 也存放 数据(data)，而 B+树只有叶子节点存放 key 和 data，其他内节点只存放 key。
  • B 树的叶子节点都是独立的;B+树的叶子节点有一条引用链指向与它相邻的叶子节点。
  • B 树的检索的过程相当于对范围内的每个节点的关键字做二分查找，可能还没有到达叶子节点，检索就结束了。而 B+树的检索效率就很稳定了，任何查找都是从根节点到叶子节点的过程，叶子节点的顺序检索很明显。
    目的都是为了减少IO
事务 不支持 支持
锁 支持表锁 支持行锁（建立在索引上），表锁
主键 可以没有 必须有，没有指定会默认生成一个隐藏列作为主键
索引 非聚集索引 聚集索引（主键索引，辅助索引）
外键 不支持 支持
查找表的行数 保存，直接取值 没有保存，全表扫描
全文索引 支持 5.7后支持

myisam索引实现

B+树，叶子节点存放数据记录的地址，如果找到指定的key，则取出data值，并以data值为地址，读取相应数据记录，有主索引和辅助索引的，两者在结构上没有区别，只是主索引要求key是唯一的，辅助索引的key可以重复，myisam的索引方式也称为非聚集索引，这是为了和innodb的聚集索引区分

innodb索引实现

表数据本身就是索引结构的，叶子节点保存了完整的数据记录，也就是一整条数据，这就是聚集索引，数据文件本身按主键建立索引，所以innodb要求一定要有主键，如果没有显式指定，就会自动选择一个，不然就是生成row_id作为主键。（可以理解innodb引擎的mysql，会默认有一个主键索引，叶子节点存放的是一整条数据记录），除了主键索引之外的索引就是辅助索引，辅助索引的叶子节点不仅包含索引的列同时包含主键，所以主键的大小会影响到索引。（每建立一个索引就相当于要维护一个B+树）

因此如果直接使用聚集索引查找的话搜索很快一次就可以查到数据，而使用辅助索引的话需要查询两次索引，第一次通过索引值查到数据以及对于的主键，第二次通过查到的主键到主索引中查找获取数据。不过还有一种情况就是一个索引包含或覆盖了所有需要查询的字段的值，这种情况叫做覆盖索引，不是主键索引的情况下，叶子节点存放的是主键和列值，因为需要回表通过主键在查找一次，就需要查询两次，如果要查询的列和索引是对应的话就不需要回表了，也就是只要查询一次

MyISAM 引擎中，B+Tree 叶节点的data 域存放的是数据记录的地址。在索引检索的时候，首先按照 B+Tree 搜索算法搜索索引，如果指定的 Key 存在，则取出其 data 域的值，然后以
data 域的值为地址读取相应的数据记录。这被称为“非聚簇索引”。

InnoDB引擎中，其数据文件本身就是索引文件。相比 MyISAM，索引文件和数据文件是分离的，其表数据文件本身就是按 B+Tree 组织的一个索引结构，树的叶节点data 域保存了完整的数据记录。这个索引的 key 是数据表的主键，因此 InnoDB表数据文件本身就是主索引。这被称为“聚簇索引（或聚集索引）”，而其余的索引都作为辅助索引，辅助索引的 data 域存储相应记录主键的值而不是地址，这也是和MyISAM 不同的地方。在根据主索引搜索时，直接找到 key 所在的节点即可取出数据；在根据辅助索引查找时，则需要先取出主键的值，在走一遍主索引。
因此，在设计表的时候，不建议使用过长的字段作为主键，也不建议使用非单调的字段作为主键，这样会造成主索引频繁分裂。

最左前缀匹配原则

MySQL建立联合索引的规则是这样的，它会首先根据联合索引中最左边的、也就是第一个字段进行排序，在第一个字段排序的基础上，再对联合索引中后面的第二个字段进行排序，依此类推。

综上，第一个字段是绝对有序的，从第二个字段开始是无序的，这就解释了为什么直接使用第二字段进行条件判断用不到索引了（从第二个字段开始，无序，无法走B+ Tree 索引）这也是 MySQL 在联合索引中强调最左前缀匹配原则的原因。

对于联合索引，MySQL会一直向右匹配直到遇到范围查询（> ， < ，between，like）就停止匹配。比如 a = 3 and b = 4 and c > 5 and d = 6，如果建立的是（a,b,c,d）这种顺序的索引，那么 d 是用不到索引的，但是如果建立的是（a,b,d,c）这种顺序的索引的话，那么就没问题，而且 a，b，d 的顺序可以随意调换。= 和 in 可以乱序，比如 a = 3 and b = 4 and c = 5 建立 （a，b，c）索引可以任意顺序。

如果建立的索引顺序是 （a，b）那么直接采用where b = 5 这种查询条件是无法利用到索引的

sql优化

一般是通过执行计划查看执行语句的时间，然后分析语句是否用到了索引，索引是否生效
1.避免select *，需要什么字段就查什么字段
2.一些公用的字段先放在内存里，要用到的时候直接拿，就不用每次都重新去数据库取，减少数据库交互

count(1),count(*),count(列)

  • COUNT（列）返回不为NULL的数量，COUNT（*），COUNT（1）返回包含NULL的数量

  • COUNT（*）是标准统计行数的语法，所以mysql做过优化

     MyISAM会把表的总行数单独记录下来，执行时可以直接返回，前提时不能有where条件，MyISAM是表级锁，不会有并发的行操作，所以查到的数据是准确的
InnoDB因为支持事务，大部分操作是行级锁，行可能被修改，记录会不准确。使用的是低成本的索引进行扫表，会优先选择最小的非聚簇索引扫描表，前提是不能有where和group by条件

  • COUNT（字段）进行全表扫描，判断指定字段的值是否为NULL，不为NULL则累加，性能慢（因为要对字段进行非NULL判断）

  • 这里的*不是扩展成所有列，实际上会忽略所有列而直接统计所有的行数

事务

  1. 原子性： 原子性确保动作要么全部完成，要么完全不起作用；
  2. 一致性： 执行事务前后，数据保持一致，例如A账户转钱到B账户，如果A账户的前减少了，B账户上的钱没有增加，此时的数据就是不一致的
  3. 隔离性： 并发访问数据库时，一个用户的事务不被其他事务所干扰
  4. 持久性：一个事务被提交之后。它对数据库中数据的改变是持久的，即使数据库发生故障也不应该对其有任何影响。
    mysql中事务默认时隐式事务，执行insert，update，delete时，数据库会自动开启事务，提交事务或回滚事务，是否开启隐式事务是由autocommit控制的，autocommit为ON表示开启了自动提交，如果要使用显式事务使用set autocommit=0；手动使用commit和rollback实现事务的提交和回滚，

并发事务带来的问题

1.脏读：一个事务读取到另一个事务中没有提交的数据
2.不可重复读：事务A多次读取一个数据，事务B在这个过程中，对数据进行修改并提交，导致事务A读取同一数据时，结果不一样（侧重于修改数据）
3.幻读：一个事务操作表中的记录，另一个事务添加或删除了一条数据，则第一个事务会误以为有数据没有修改，好像发生了幻觉一样（侧重于新增或删除）
不可重复读的重点是修改，幻读的重点是新增或者删除，解决不可重复读只需要锁要读取的行，幻读需要锁表

事务隔离级别

1. READ UNCOMMIT，读未提交，会产生脏读，不可重复读，幻读
即使没有提交，每个事务的操作对于其他事务也都是可见的，在这种情况下，事务可以读取未提交的数据，即脏读
2. READ COMMIT, 读已提交，会产生不可重复读，幻读
事务所修改的数据只有提交之后才会被其他事务读取到。即事务从开始到结束，所做的任何修改对其他事务是不可见的，但是执行两次同样的查询，可能会出现结果不一致的情况（查询过程中有其他事务提交完成），这一级别又叫不可重复读
3. REPEATABLE READ，可重复读，会产生幻读（mysql默认）
是mysql的默认事务隔离级别，保证了在同一个事务中多次读取同样的记录结果是一致的，理论上可重复读没法解决幻读的问题。幻读是指在某个事务读取某个范围内的记录时，另外一个事务又再次在该范围内插入了新的记录，当之前的事务再次读取该范围的记录时，就会产生幻读，不过innodb通过多版本并发控制MVVC解决了这个问题
4. SERIALIZABLE,可串行化
最高的隔离级别。通过强制让事务串行执行，可以避免所有的问题，本质上，serializable会在读取的每一行数据上都加上锁，对性能影响严重。

数据库表设计

1.关注表字段的扩展性，例如性别这个字段，正常可以用bool类型，true代表男，false代表女，如果有未知的情况就无法表示了，因此可以用int类型或者枚举类型代替
2.考虑表的可维护性和可读性，例如主表有奖品信息的字段，如果奖品有多种类型，还有多种类型，多个奖品，就应该独立出一张表通过字段关联来存放奖品信息而不是放在主表中

innodb行锁

innodb有行锁和表锁，行锁是基于索引，所以查询的时候如果不是用索引字段查询的话使用的还是表锁

innodb实现了两种类型的行锁：

  1. 共享锁（S）：加了锁的记录，所有事务都能去读取但不能修改，同时阻止其他事务获得相同数据集的排他锁；

  2. 排他锁（X）：允许已经获得排他锁的事务去更新数据，阻止其他事务取得相同数据集的共享读锁和排他写锁；

innodb的表锁

也叫意向锁，在加表锁的时候，需要遍历该表所有的记录，判断是否有排他锁，来检测表锁和行锁的冲突，分为：

  1. 读意向锁（IS锁）

  2. 写意向锁（IX锁）

当事务要在记录上加上行锁的时候，先在表上加上意向锁，这样判断表中是否有记录正在加锁，只要查看表上是否有意向锁就可以了，意向锁之间是不会产生冲突的，它只会阻塞表级读锁或写锁。

innodb加锁方法

  1. 意向锁是自动加的，不需要用户干预

  2. 对于update，delete，insert语句，innodb会自动给涉及的数据集加上排他锁

  3. 对于普通的select语句，innodb不会加锁，在开启事务的情况下可以显式给记录集添加共享锁或排他锁

    select for update

在执行这个 select 查询语句的时候，会将对应的索引访问条目加上排他锁（X锁），也就是说这个语句对应的锁就相当于update带来的效果；

使用场景：为了让确保自己查找到的数据一定是最新数据，并且查找到后的数据值允许自己来修改，此时就需要用到select for update语句；

性能分析：select for update语句相当于一个update语句。在业务繁忙的情况下，如果事务没有及时地commit或者rollback可能会造成事务长时间的等待，从而影响数据库的并发使用效率。

select lock in share mode

in share mode 子句的作用就是将查找的数据加上一个share锁，这个就是表示其他的事务只能对这些数据进行简单的 select 操作，而不能进行 DML 操作。

使用场景：为了确保自己查询的数据不会被其他事务正在修改，也就是确保自己查询到的数据是最新的数据，并且不允许其他事务来修改数据。与select for update不同的是，本事务在查找完之后不一定能去更新数据，因为有可能其他事务也对同数据集使用了 in share mode 的方式加上了S锁；

性能分析：select lock in share mode 语句是一个给查找的数据上一个共享锁（S 锁）的功能，它允许其他的事务也对该数据上S锁，但是不能够允许对该数据进行修改。如果不及时的commit 或者rollback 也可能会造成大量的事务等待。

drop，delete，truncate

都表示删除

1.drop，truncate只删除表的数据不删除表的结构

2.速度：drop>truncate>delete

3.delete执行完后要提交事务才会生效

使用

1.不需要表的使用，用drop

2.删除部分数据，用delete，并且带上where子句

3.保留表删除全部数据，用truncate
