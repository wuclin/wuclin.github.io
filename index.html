数据库部分

数据库三大范式

1.第一范式（1NF）：列不可再分
每一列都是不可再分的属性，确保每一列的原子性，比如地址可以拆分为省，市
2.第二范式（2NF）：属性完全依赖于主键
在第一范式的基础上，每一行数据可以被区分，通常是加多一个列，也就是主键，主要是针对于联合主键的情况，要变成只有一个主键
3.第三范式（3NF）：属性不依赖于其他非主属性，直接依赖于主键
即数据不能存在传递关系，每个属性都和主键有直接关系而不是间接关系，非主键外的所有字段必须互不依赖

线程交互

Posted on 2022-06-08 In Java

同步队列和等待队列

对应线程的两个状态，阻塞和等待

阻塞：因为synchronized没获取到锁的线程就会阻塞在同步队列，同步队列中放的是尝试获取锁但失败的线程

等待：使用了wait方法，就会进入到等待队列，等待进入同步队列去竞争同步锁，进入等待队列之后，需要notify唤醒或者使用wait(long timeout)等待指定时间后唤醒

等待队列中会有多个线程wait同一个对象的情况，此时如果某一个线程调用了对象的notify方法，会任意的唤醒等待队列中的一个线程

批量插入的方式

1.使用for循环插入，这样速度很慢，数据量小的时候可以使用

2.使用mybatis的foreach，这种情况下其实就是拼接sql，因为mysql本来就是支持批量插入的，有个问题就是拼接的sql不能太长，超过mysql服务器中的max_allow_packet的话就会报错（关于这里有两种写法，一种是在values里面使用foreach，另外一种是在insert使用foreach，最好是使用第一种写法；因为第二种写法，首先是字符长度增加，会导致每个批次可插入的数量变少，第二这种形式的插入其实是执行多条insert语句，这会导致如果后面出错了，前面的语句就会已经插入了，由于数据库默认是单条插入语句就会提交的，出错之前都是单条插入，所以没有回滚数据，如果想在插入失败回滚的话，需要使用spring事务，通过手动抛出异常的方式来回滚，而第一种方法由于是一条sql，所以就不需要使用事务）

3.使用jdbc的batch批处理

tips:批量插入要在数据库的配置加上allowMultiQueries=true

抽取大数据表

Posted on 2022-06-07

在项目中遇到抽取大数据并进行处理的需求（迁移数据，批量处理数据），解决过程，

  1. 使用limit分页查找的方式，部分查找数据

    对于大量的数据，比如200w的数据量，一般不会一次性加载到内存中，一般会采取分页的方式抽取数据，由于limit是在查询结果的基础上跳过多少条记录，也就是跳过多少条记录也是需要记录的。当查询的起始记录页码越来越大的时候，速度也会变慢。
    优化方式：

    即使使用以上的方式，在进行大量数据操作时速度也是很慢，比如读取1000000条数据，再对数据进行处理，这里的耗时就很长

  2. 利用游标查找：以io的形式抽取数据，速度最快，会以一条一条数据的形式抽取数据到内存中。

  3. 利用流式查找：使用游标可以不用一次性取出所有的数据到内存中，而是根据fetchSize的大小来取数据，在使用的时候需要保持数据库连接，而mapper方法通常执行完后连接就关闭了，因此cursor也会关闭，要解决这个问题就要保持数据库连接的打开，有两种方式：1.在controller层手动打开连接；2.加上@Transactional注解

  4. 以上的方式都是在处理数据的过程中将数据保存在jvm或者内存中，还有一种方式是将数据存放在数据库的临时表空间中,使用以下的sql抽取表的数据，一方面是依赖于数据库的临时表空间大小，另一方面是会产生锁表的问题

    在默认的事务隔离级别下，对于语句insert into t1 select * from t2 加锁的规则是：t1为表锁，t2每扫描一条数据就会锁住一条数据，在这个过程中相当于是对t2也是加了表锁，如果要避免全表扫描的话，就需要给条件字段加索引，由于走了索引就不会有缩表的情况，不过一般这种方法用于复制表或者转移表数据

悲观锁

悲观锁在操作数据时比较悲观，认为别人会同时修改数据。因此操作数据时直接把数据锁住，直到操作完成后才会释放锁；上锁期间其他人不能修改数据。

对于synchronized和ReentrantLock都是独占锁，ReentrantReadWriteLock里的写锁式独占锁，读锁是共享锁
